# 화면 구성

이 문서는 합의된 전역 화면 구조와 아직 구체화 중인 화면별 책임을 정리합니다.

## 앱 수준 화면

### 최초 연결 화면

워크스페이스가 연결되지 않은 최초 실행 또는 연결 복구가 필요할 때만 표시합니다.

- GitHub 로그인
- 기존 OKF 지식 저장소 연결
- 새 OKF 지식 저장소와 `.okf/workspace.yml` 초기화
- 저장소 접근 권한과 워크스페이스 설정 검증

최근 워크스페이스 목록, 워크스페이스 선택과 전환 화면은 제공하지 않습니다. 연결이 완료된 뒤 앱을 다시 열면 현재 워크스페이스의 Home으로 바로 진입합니다.

### 워크스페이스 설정

- 현재 OKF 지식 저장소와 `.okf/workspace.yml`
- 백엔드·프론트엔드 코드 저장소
- GitHub 인증과 동기화
- 기본 브랜치
- 제품 구조와 문서 템플릿
- Codex/MCP 연결 안내
- 연결 오류 복구와 현재 워크스페이스 재연결

프로젝트 저장소 연결은 워크스페이스의 좌측 주 메뉴에 두지 않습니다. Settings는 현재 단일 워크스페이스의 연결을 관리하며 다른 워크스페이스로 빠르게 전환하는 선택기를 제공하지 않습니다.

## 워크스페이스 전역 구조

**상태: 합의**

```text
Home
Documents
Project
────────
Settings
```

- `Work`는 독자적인 데이터와 행동이 없어 전역 메뉴에서 제외합니다.
- `Activity`는 Home의 타임라인으로 흡수합니다.
- `Reviews`는 독립 콘텐츠가 아니라 문서나 PR에 붙는 처리 과정이므로 전역 메뉴에서 제외합니다.
- 사용자가 확인해야 할 검토, 결정과 작업 시작 알림은 Home에서 모아 보여주고, 실제 처리는 원래 문서, Issue나 PR 문맥에서 진행합니다.
- 사이드바 상단에는 현재 프로젝트 이름을 정적인 식별 정보로 표시하며 드롭다운 화살표나 워크스페이스 선택 동작을 두지 않습니다.
- 좌측 사이드바는 완전히 접을 수 있으며, 접으면 본문이 남은 폭을 사용합니다. 펼친 상태의 토글은 사이드바 최상단 오른쪽에 두고, 접힌 상태에서는 같은 기능의 버튼을 본문 왼쪽 위에 표시합니다.
- 사이드바 토글은 Lucide의 `PanelLeftClose`·`PanelLeftOpen` 계열 아이콘과 설명 tooltip을 사용하고, macOS `Command + Backslash`, Windows `Ctrl + Backslash` 단축키를 제공합니다.

### Home

앱을 열었을 때 현재 단일 워크스페이스의 프로젝트 진행 상황과 다음 행동을 보여줍니다.

- `프로젝트 진행 상황` 제목
- 제목 아래의 인사말과 오늘 날짜
- 열린 Issue, 진행 중인 Issue, 이번 주 완료 Issue 요약
- 기능별 진행률과 완료 Issue 수
- 내가 확인할 항목
  - 검토 필요
  - 결정 필요
  - 작업 시작 가능
  - 응답 필요
- 발생 시간을 포함한 최근 활동

기능 진행률은 초기에는 연결된 Issue의 `완료 수 / 전체 수`로 계산하고 퍼센트와 실제 개수를 함께 보여줍니다. 작업 크기를 반영한 가중 진행률은 초기 범위에서 제외합니다.

백엔드 API 완료처럼 누군가의 승인이 필요하지 않더라도 후속 작업자가 다음 행동을 시작해야 하는 이벤트는 `작업 시작 가능` 항목으로 표시합니다. 연결된 선행 Issue 종료나 PR 병합을 기준으로 만들며, 정확한 연결이 없으면 자동으로 추론하지 않습니다.

현재 합의한 것은 Home의 정보 구성, 표시 순서와 영역별 책임입니다. 색상, 타이포그래피, 간격, 카드 모양과 반응형 표현을 포함한 최종 시각 디자인은 별도 단계에서 확정합니다.

### Documents

- 제품 구조 탐색
- 문서 라이브러리
- 문서 편집과 렌더링
- 문서별 Git history
- 관련 분석·테스트·API·코드
- 현재 버전과 제안 버전 전환

Documents는 문서가 중심인 공간입니다. `History`는 별도 전역 메뉴가 아니라 각 문서의 커밋·변경·승인 이력을 보여주는 상세 보기입니다. 진행 중인 작업을 별도 목록으로 소유하지 않으며, 문서에서 관련 Issue나 리뷰 상태를 참조 정보로 확인할 수 있습니다.

문서 상세는 기본적으로 읽기 모드로 엽니다. 기본 브랜치의 확정 문서에서 `편집`을 누르면 변경 제안을 만들고, 이미 작업 브랜치나 PR 버전을 보고 있다면 해당 버전을 바로 편집합니다.

상세 화면은 다음 3개 영역을 사용합니다.

```text
문서 트리 | 읽기·리치 편집·Markdown 원문 캔버스 | 접을 수 있는 문맥 패널
```

- 본문 캔버스는 한 번에 읽기, 리치 편집 또는 원문 중 하나에 집중합니다.
- 우측 문맥 패널은 `목차 / 연결 / History / 댓글` 탭으로 구성합니다.
- `연결`에는 관련 Issue, PR, 코드와 API를 모아 보여줍니다.
- `댓글`은 변경 제안이나 리뷰 문맥에서만 표시합니다.
- 문서 제목 아래에는 `확정본 · main`과 같이 현재 Git 버전을 보여주는 선택기를 두고, 열린 변경 제안이 있으면 선택기 옆에 개수 배지를 표시합니다.
- 마지막 수정 시각과 작성자는 제목 아래에 짧게 표시하고, 커밋별 상세 수정 이력과 diff는 우측 `History` 탭에서 확인합니다.

문서 제목과 Git 파일 경로는 서로 다른 값으로 다룹니다. 화면에 보이는 제목은 OKF frontmatter의 `title`이며 편집 모드에서 수정합니다. 파일명과 폴더를 바꾸는 구조 작업은 오버플로 메뉴의 `파일 경로 변경…`으로 제공하고, 영향받는 문서 링크를 확인한 뒤 변경 제안에 반영합니다. 의미가 모호한 `이름 변경` 액션은 사용하지 않습니다.

문서 헤더의 주요 작업은 텍스트 `편집` 버튼으로 노출하고, 그 옆에 Lucide `Ellipsis`를 사용한 가로 점 오버플로 메뉴를 둡니다. 오버플로 메뉴는 다음 순서로 구성합니다.

1. 파일 경로 변경, 복제
2. 링크 복사, 파일 경로 복사, GitHub에서 보기
3. 구분선 아래의 `삭제 제안…`

확정 문서의 삭제는 즉시 실행하지 않고 삭제 변경 제안을 만듭니다. 위험 액션은 메뉴 마지막에 `Trash2` 아이콘과 danger 스타일로 표시하며, 대상 문서와 영향을 받는 링크를 설명하는 확인창을 거칩니다.

### Project

- 현재 단일 워크스페이스에 연결된 GitHub Project
- Project의 Board와 List
- 전체 Issue와 나에게 할당된 Issue
- Iteration, 사용자 정의 필드와 필터
- Issue 생성, 수정, 댓글과 종료
- 관련 기능·시나리오·처리 케이스·문서 표시
- 연결된 PR과 코드 확인

기본 정책은 `OKF 지식 저장소 1개 = 워크스페이스 1개 ↔ GitHub Project 1개`입니다. Project는 연결된 여러 저장소의 Issue와 PR을 모아 진행 상황을 보여주는 화면이고, 실제 구현 작업의 원본은 GitHub Issue입니다. 초기에는 추적 가능한 실제 Issue와 PR만 Project 항목으로 사용하고 Draft Issue 지원은 제외합니다.

작업 기간은 저장소별 GitHub Milestone이 아니라 GitHub Project의 `Iteration` 필드를 사용합니다. Project 화면은 현재 Iteration을 기본으로 보여주고 지난·다음 Iteration으로 전환할 수 있습니다.

Project 상단에는 프로젝트 이름과 Iteration 선택만 표시합니다. 진행률 요약은 Home의 책임으로 두고 Project에는 진행 막대, 퍼센트와 완료 개수를 중복 표시하지 않습니다. 별도 규칙이 없는 `On track`, `막힘`, `미할당` 요약도 표시하지 않습니다. 용도가 확정되지 않은 상단 더보기 메뉴와 전역 `새 Issue` 버튼도 두지 않습니다.

새 Issue는 현재 Iteration의 `Backlog` 컬럼 하단에서 생성합니다. 생성 UI는 기존 Issue와 같은 크기의 추가 카드이며, 카드 중앙에 `+ 새 Issue` 버튼을 표시합니다. 생성된 Issue는 `Backlog` 상태로 시작하며 이후 Board에서 다른 상태로 이동합니다. 다른 컬럼에는 반복적인 추가 버튼을 두지 않고, 종료된 지난 Iteration에서는 Issue를 추가하지 않습니다.

사용자 시나리오 전체 진행률은 Home에서 관련 문서 상태와 Project의 Issue를 조합해 보여줍니다. `Issues`를 별도 전역 메뉴로 중복 제공하지 않으며, Issue 상세는 Project 문맥 안에서 엽니다.

Board에서 Issue를 선택하면 우측 패널로 상세를 열어 빠르게 확인하고 처리합니다. 긴 본문, 문서, 코드나 diff를 넓게 봐야 할 때는 동일한 Issue를 전체 화면으로 확장할 수 있습니다.

Issue 상세 패널은 탭으로 내용을 숨기지 않고 다음 순서의 단일 스크롤로 구성합니다.

1. 저장소, Issue 번호, 제목, 상태와 담당자
2. 설명
3. 완료 조건 또는 체크리스트
4. 담당자, Iteration, Label과 저장소 속성
5. 연결된 문서, 코드와 PR
6. 댓글과 댓글 작성

우측 패널에서는 빠른 확인, 댓글과 상태 변경을 처리하고, 전체 화면으로 확장하면 같은 정보를 더 넓게 읽습니다. 상세 화면은 GitHub Issue의 개념과 동작을 새로 만들기보다 GitHub를 공유 원본으로 유지한 채 Hub 안에서 사용할 수 있게 옮겨 놓는 역할을 합니다. Hub만의 책임은 Issue와 문서·코드·PR·제품 구조의 연결을 한 문맥에서 보여주는 것입니다.

현재 승인한 것은 Issue 상세의 진입 방식과 정보 순서입니다. 시안에서 깨져 보이는 간격, 크기, 정렬과 반응형 표현은 최종 디자인 시스템 단계에서 다시 조정합니다.

### 문맥 안의 리뷰

- Documents의 변경 제안에서 문서·Mermaid diff와 댓글을 확인합니다.
- Issue 또는 PR에서 코드 diff, 댓글, 승인과 수정 요청을 처리합니다.
- 선택형 결정은 대상 문서나 PR의 맥락 안에서 처리합니다.
- Home의 확인할 항목에서 해당 문서, Issue나 PR로 이동합니다.

### API Catalog

OpenAPI를 원본으로 사용할 경우 Documents 안에 API 탐색 화면을 제공합니다.

- 서비스 또는 태그별 API 목록
- Path, method와 `operationId` 검색
- 요청·응답 스키마와 오류 계약
- 관련 기능과 처리 케이스
- 변경 제안과 버전 diff

API를 제품 구조 트리에 직접 중첩하지 않습니다.

## 공통 작업 화면

문서, 코드와 리뷰를 동시에 다루는 화면은 다음 영역을 조합할 수 있습니다.

```text
현재 문맥 탐색 | 문서 또는 리뷰 본문 | 관련 코드·API·변경 정보
```

- 좌측 영역은 항상 고정된 전체 트리가 아니라 현재 작업에 필요한 탐색을 제공합니다.
- 우측 보조 영역은 관련 코드, diff, 댓글 또는 속성으로 전환됩니다.
- 문서와 코드는 같은 화면에서 읽되 코드를 직접 편집하지 않습니다.

## 현재 UI 기술 후보

### 진행 순서

1. Home, Documents, Project, Settings와 문맥형 리뷰 화면의 정보 구성을 확정합니다.
2. 화면 간 이동과 주요 사용자 흐름을 확정합니다.
3. 전체 화면에서 반복되는 패턴을 기준으로 디자인 시스템을 확정합니다.
4. 확정된 디자인 시스템으로 고해상도 화면을 설계합니다.
5. 화면과 기술 명세가 승인된 뒤 구현 계획을 작성합니다.

화면 구성을 정하는 동안 만든 시안은 정보의 우선순위와 영역 배치를 검증하기 위한 것입니다. 시안의 개별 컴포넌트 모양을 최종 디자인으로 간주하지 않습니다.

### 합의된 방향

- Tauri + 웹 프론트엔드
- shadcn/ui 기반 컴포넌트
- 일반 화면은 여백이 있는 디자인, 트리·코드·테이블은 compact 밀도
- 문서 편집은 MDXEditor 계열 리치 에디터
- Markdown 원문과 diff는 CodeMirror
- 기본 서체는 `Pretendard Variable`을 사용하고 `Pretendard`와 시스템 sans-serif를 fallback으로 둡니다.
- 데스크톱 앱에 Pretendard 웹폰트 파일을 포함해 오프라인에서도 동일하게 렌더링합니다.
- 인터페이스 아이콘은 Lucide를 사용합니다.
- 의미가 불명확한 장식용 아이콘을 두지 않고, 아이콘 단독 버튼에는 이름 또는 tooltip을 제공합니다.
- 현재 시안의 중립 회색 surface, indigo primary와 상태별 green·amber·blue·red 색상 방향을 유지합니다.

이전에 사용한 `Nova`는 디자인 스타일 후보 이름이며 프로젝트명이 아닙니다.

Home 시안 우측 상단의 클로버 모양은 기능이 정해지지 않은 임시 표시였으므로 제거합니다. 우측 상단에는 현재 동기화와 사용자 메뉴만 두고, 실제 전역 기능이 확정될 때만 아이콘을 추가합니다.

### 미결정

- Documents의 Library·Structure·History·APIs 하위 탐색 방식
- 제품 구조 트리를 항상 표시할지 필요할 때만 표시할지
- Home의 확인할 항목에서 문서·Issue·PR 화면으로 전환하는 방식
- Project의 Board·List·My Issues·Iterations 하위 탐색 방식
- 정확한 색상·타이포그래피·간격 토큰과 컴포넌트 variant
- Lucide 아이콘별 제품 의미와 기본 크기·stroke 규칙
