# 정보 모델

모든 정보를 하나의 트리에 넣지 않고 서로 다른 성격의 정보를 분리합니다.

## 세 가지 핵심 구조

| 구조 | 역할 | 예시 |
|---|---|---|
| 제품 구조 | 사용자가 만들고 있는 기능의 의미 계층 | 기능, 사용자 시나리오, 처리 케이스 |
| 문서 자료 | 여러 대상에 연결될 수 있는 지식과 근거 | 설계, 분석, 테스트, 실험, 운영 문서 |
| 변경과 리뷰 | 아직 확정되지 않은 제안과 대화 | 브랜치, PR, 댓글, 승인, 수정 요청 |

API, 코드, GitHub Issue/PR과 커밋은 외부 참조로 연결합니다.

## 제품 구조 트리

### 기존 후보

```text
기능 그룹
└─ 기능
   └─ 사용자 시나리오
      └─ 처리 케이스
```

4단계가 항상 필요한 것은 아니며, 기능이 많아질수록 탐색이 깊어질 수 있다는 우려가 있습니다. 따라서 현재 확정된 구조가 아닙니다.

### 검토할 대안

#### A. 2단계

```text
기능
└─ 처리 케이스
```

단순하지만 사용자 흐름과 구현 분기를 구분하기 어렵습니다.

#### B. 기본 3단계

```text
기능
└─ 사용자 시나리오
   └─ 처리 케이스
```

현재 가장 유력한 후보입니다. `기능 그룹`은 트리 단계가 아니라 컬렉션, 태그 또는 상위 필터로 처리할 수 있습니다.

#### C. 선택적 4단계

기본은 3단계로 사용하고, 규모가 큰 워크스페이스에서만 기능 그룹을 활성화합니다.

### 커스텀 정책 후보

- 내부 의미 타입은 제한된 집합으로 유지합니다.
- 팀은 화면 표시명을 변경할 수 있습니다.
- 필요하지 않은 단계는 생략할 수 있습니다.
- 같은 타입의 무한 중첩은 허용하지 않습니다.
- 전체 트리 대신 현재 작업, 최근 항목, 즐겨찾기와 필터를 주 탐색 경로로 제공합니다.

## Documents

문서는 제품 구조의 자식이 아니라 독립 자료입니다. 하나의 문서가 여러 기능, 처리 케이스, API와 코드에 연결될 수 있습니다.

### 공통 필드 후보

```yaml
id: stable-document-id
type: team-defined-type
status: draft | proposed | accepted | archived
title: 문서 제목
related:
  features: []
  scenarios: []
  cases: []
  api_operations: []
  code: []
  issues: []
  pull_requests: []
  commits: []
```

필드명과 상태 목록도 최종 스키마가 아니라 논의를 위한 예시입니다.

## 문서 타입 정책

문서 타입은 팀과 프로젝트 성격에 따라 달라질 수 있어 고정 목록으로 확정하지 않습니다.

### 후보 타입 예시

- 기능 또는 처리 흐름 설계
- API/DTO/예외 계약을 설명하는 보조 문서
- 성능·병목 분석
- 실험 계획과 결과
- 테스트 계획과 테스트 결과
- 기술 조사
- 장애 회고
- 운영 가이드

### 검토할 방식

1. 완전 고정 타입: AI와 자동화는 쉽지만 팀별 요구를 반영하기 어렵습니다.
2. 완전 자유 타입: 유연하지만 의미가 중복되고 검색과 자동화가 불안정합니다.
3. 기본 타입 템플릿 + 팀 커스텀: 현재 추천 후보입니다.

기본 타입에는 공통 필드와 권장 섹션을 제공하고, 팀은 이름과 추가 필드를 확장할 수 있습니다. 어떤 기본 타입이 실제로 필요한지는 사례 조사 후 결정합니다.

## 확정된 지식과 변경 제안

```text
기본 브랜치의 문서
    │ 현재 합의된 문서
    │
    ├─ 변경 제안 브랜치
    │      ├─ 수정된 문서
    │      ├─ 수정된 Mermaid
    │      └─ 변경 이유
    │
    └─ GitHub PR
           ├─ 댓글
           ├─ 선택
           ├─ 승인 또는 수정 요청
           └─ merge 후 기본 브랜치 반영
```

리뷰 중인 정보도 Git에는 존재하지만 기본 화면의 현재 문서로 취급하지 않습니다.

## 문서 history

- 파일 경로와 안정적인 문서 ID로 이력을 추적합니다.
- 특정 문서의 커밋 목록과 변경자를 보여줍니다.
- Markdown diff와 렌더링 결과를 함께 제공합니다.
- Mermaid는 이전 버전과 제안 버전을 각각 렌더링해 비교할 수 있습니다.
- 관련 PR, Issue와 결정 결과를 함께 표시합니다.

## 결정과 리뷰 활동의 구분

모든 댓글이나 승인을 영구 결정 문서로 만들지 않습니다.

| 활동 | 보존 방식 |
|---|---|
| 질문과 일반 댓글 | GitHub 리뷰 대화 |
| 오탈자와 작은 수정 | 문서 Git history |
| 계약·동작·구조를 확정한 결과 | 문서 본문 또는 장기 결정 기록 |
| 이전 결정을 변경한 결과 | 이전 기록과 연결된 새 변경 이력 |
