# 미결정 사항과 조사 과제

이 문서는 아직 확정하지 않은 제품 결정을 추적합니다. 아래 항목이 해결되기 전까지 관련 내용을 최종 요구사항으로 간주하지 않습니다.

## 우선순위 1. 제품 구조 트리 깊이

### 질문

기능이 많아져도 이해하기 쉬운 기본 계층은 몇 단계인가?

### 후보

- 2단계: 기능 → 처리 케이스
- 3단계: 기능 → 사용자 시나리오 → 처리 케이스
- 선택적 4단계: 기능 그룹을 필요한 워크스페이스에서만 활성화

### 현재 추천 후보

기본 3단계를 사용하고 기능 그룹은 컬렉션 또는 필터로 처리합니다. 실제 예시 여러 개를 넣어 탐색성과 진행률 표현을 비교해야 합니다.

## 우선순위 2. 문서 타입

### 질문

AI와 자동화가 이해할 수 있을 만큼 일관되면서 팀별 업무 방식도 수용하려면 타입을 얼마나 제한해야 하는가?

### 조사할 것

- 실제 개발 과정에서 반복되는 문서 사례 수집
- 설계, 분석, 실험, 테스트, 회고와 운영 문서의 공통 필드
- 타입별 권장 섹션과 검증 규칙
- 사용자 정의 타입의 이름 충돌과 마이그레이션 방식
- 팀 템플릿을 Git에 저장하고 공유하는 방식

### 현재 추천 후보

소수의 기본 타입 템플릿을 제공하고 팀이 이름, 섹션과 추가 필드를 확장하도록 합니다. 기본 타입 목록은 사례 조사 전 확정하지 않습니다.

## 우선순위 3. 대표 유스케이스 상세화

개발 중 확인이 필요한 상황을 첫 번째 수직 흐름으로 설계할지 결정해야 합니다.

- 변경 제안 생성 시 필요한 최소 정보
- 문서 변경 없이 질문만 하는 경우
- 변경 제안에 여러 문서와 코드가 포함되는 경우
- 응답 지연과 알림
- 수정 요청과 재검토
- 승인 뒤 문서·Issue·구현 상태 갱신

## 우선순위 4. API 계약 원본

- 현재 백엔드 API 문서화 방식 확인
- OpenAPI 파일의 저장소 위치
- 설계 우선과 코드 우선 방식의 관계
- Postman Collection 자동 생성·동기화 범위
- Hub에 API 호출 기능을 포함할지

## 우선순위 5. 리뷰 데이터 모델

- 일반 질문과 문서 변경 PR의 구분
- 선택형 응답을 GitHub에 저장하는 방식
- 문서 블록 댓글과 GitHub 라인 댓글 매핑
- 리뷰가 닫힌 뒤 대화의 검색과 보존
- 영구 결정 기록으로 승격하는 기준

## 우선순위 6. 작업 모델

- 사용자 시나리오와 처리 케이스가 Issue와 어떤 관계인지
- GitHub Project의 상태·사용자 정의 필드와 Hub 상태를 어떻게 매핑할지
- 오프라인 초안과 GitHub 동기화 충돌 처리
- 배포 상태를 어떤 GitHub 데이터에서 가져올지

워크스페이스당 하나의 GitHub Project를 연결하고 실제 작업은 GitHub Issue를 원본으로 사용하는 구조는 합의했습니다. 초기 Project 항목은 실제 Issue와 PR로 제한합니다.

## 우선순위 7. 화면별 세부 구성

- Documents에서만 제품 구조 트리를 확장할지
- Project의 Board·List·My Issues·Milestones 하위 탐색
- 문서·코드·리뷰 패널을 동시에 여는 방식
- Home의 확인할 항목에서 대상 문서·Issue·PR로 이동하는 방식

전역 메뉴는 `Home / Documents / Project / Settings`로 합의했습니다. Activity는 Home으로 흡수하고, Issues는 Project 안의 실제 작업 항목으로 다루며, Reviews는 대상 문서나 PR의 문맥 안에서 처리합니다.

Home의 기본 배치는 합의했습니다. 상단에 Issue 요약을 두고, 기능 진행률, 내가 확인할 항목과 시간이 표시된 최근 활동을 이어서 보여줍니다. 세부 카드 크기와 반응형 배치는 화면 시안에서 조정합니다.

## 우선순위 8. 이름

- OKF를 이름에 직접 포함할지
- 디자인 스타일 후보명과 제품명을 명확히 구분할 것
