Approved product workflow

문서를 만들고, 함께 검수하고, 확정하기까지

Hub는 별도의 협업 DB를 만들지 않습니다. 문서와 버전은 Git에, 작업과 편집자는 GitHub Issue에, 검수와 승인은 GitHub PR에 남기면서 사용자는 Hub 한 화면에서 전체 흐름을 처리합니다.

로컬 우선 DB 없음 Git + GitHub 문서 단위 소프트 잠금 Draft PR 반복 검수
00 · Overview

전체 흐름부터 한 번에 보기

새 문서와 기존 문서는 시작 방식만 다르고, 편집 이후에는 하나의 검수·병합 흐름으로 합쳐집니다.

문서 변경의 생명주기왼쪽에서 오른쪽으로 진행

확정본

main의 문서만 팀이 현재 합의한 지식입니다.

변경 제안

브랜치와 worktree에서 편집하며, 기본 문서를 바로 덮어쓰지 않습니다.

검수 기록

댓글, 선택형 결정, 승인과 수정 요청은 같은 PR 안에 이어집니다.

01 · Identity

문서는 경로가 아니라 안정적인 ID로 찾는다

제목이나 파일 위치가 바뀌어도 같은 문서임을 알아야 잠금과 이력을 안전하게 이어갈 수 있습니다.

불변

문서 ID

새 문서를 만들 때 기기에서 UUID v4를 생성하고 doc_ 접두사를 붙입니다. 이동하거나 제목을 바꿔도 유지됩니다.

변경 가능

제목과 파일 경로

제목은 문서 내용의 속성이고, 경로는 Git의 위치입니다. 복제할 때는 새 ID를 발급하며, ID가 없는 기존 문서는 첫 편집 때 추가합니다.

  • 잠금 비교: 문서 ID
  • 화면 표시·경로 충돌 확인: 현재 경로
  • 이력: ID → 과거·현재 경로 인덱스
02 · Start

변경 시작: 새 문서와 기존 문서를 구분한다

기존 문서는 충돌을 예방하기 위해 Issue 잠금을 먼저 확보하고, 새 문서는 개인 초안으로 바로 시작합니다.

변경 시작 판단기존 문서만 Issue가 필수
새 문서

개인 초안으로 시작

  • 생성자가 바로 편집합니다.
  • 브랜치는 소유자와 change ID를 포함합니다.
  • 단순 검수는 Issue 없이 Draft PR을 만들 수 있습니다.
  • 다른 개발자에게 편집권을 넘길 필요가 생기면 Issue를 추가합니다.
기존 문서

Issue로 소프트 잠금

  • okf-document-change 라벨을 가진 열린 Issue가 잠금입니다.
  • Assignee 한 명이 현재 편집자입니다.
  • 잠금을 확보한 다음에만 branch와 worktree를 만듭니다.
  • 다른 팀원은 읽기, 댓글, 결정 응답과 검수는 가능합니다.
Issue에 어떤 구조화 정보를 남기는가
사람이 읽는 설명 아래에 Hub가 읽는 숨김 메타데이터를 둡니다. 한 Issue가 관련 문서 여러 개를 포함할 수 있지만 현재 편집자는 한 명입니다.

03 · Edit

편집과 저장: 로컬 저장과 팀 공유를 분리한다

입력할 때마다 원격에 커밋하지 않습니다. 로컬 자동저장으로 작업을 보호하고, 의미 있는 체크포인트를 push해 다른 기기와 팀원이 볼 수 있게 합니다.

1

독립된 worktree에서 편집

한 변경 작업마다 별도 폴더와 working tree를 사용해 다른 변경의 수정 상태와 섞이지 않게 합니다.

Git 결과브랜치는 커밋 이력을 가리키고, worktree는 그 브랜치를 편집하는 로컬 파일·index·HEAD를 가집니다.
2

입력은 로컬 자동저장

Hub 에디터가 현재 worktree의 파일에 저장합니다. 이 단계는 네트워크가 없어도 동작합니다.

화면 상태저장됨 현재 기기에는 안전하지만 아직 팀 공유 상태는 아닙니다.
3

의미 있는 지점에서 체크포인트

수동 동기화, 일정한 작업 구간 또는 검수 요청 직전에 변경을 commit하고 원격 브랜치에 push합니다.

화면 상태동기화 중 → 동기화됨 / 실패
4

main 변경을 가져올 때는 merge

공유 중인 draft 브랜치의 커밋 해시를 다시 쓰지 않도록 rebase 대신 merge로 최신 main을 반영합니다.

공유 안전성이미 push한 브랜치의 history를 강제로 다시 쓰지 않습니다.

stash는 체크포인트가 아닙니다. stash는 현재 기기의 로컬 임시 보관이므로 다른 기기 복구와 팀 공유를 보장하지 않습니다.

04 · Handoff

같은 문서를 고치고 싶다면 편집권을 요청한다

같은 문서는 worktree가 달라도 한 시점에 한 명만 편집합니다. 편집권 이전이 완료되기 전에는 요청자가 별도 worktree에서 수정할 수 없으며, 서로 다른 문서만 각자의 worktree에서 동시에 편집할 수 있습니다.

편집권 넘기기

같은 변경을 이어서 편집

  1. 현재 편집자가 checkpoint commit·push
  2. 원격 동기화와 clean 상태 확인
  3. 현재 기기의 기존 worktree 정리
  4. Issue Assignee를 요청자로 변경
  5. 요청자가 같은 브랜치로 새 worktree 생성
  6. 기존 편집자는 읽기·검수 모드로 전환
검수자로 참여

직접 편집하지 않고 제안

현재 편집자는 유지하고 상대방이 PR 댓글, 문서 블록 의견 또는 결정 응답을 남깁니다.

요청 거절

이유를 남기고 계속 편집

거절 이유는 구조화된 Issue 댓글로 남아 요청자와 현재 편집자가 같은 맥락을 봅니다.

편집권 이전은 순차적으로 완료되어야 합니다. checkpoint push나 기존 worktree 정리에 실패하면 Assignee를 변경하지 않습니다. 이전이 끝나면 같은 Issue·브랜치·PR은 유지하지만, worktree는 공유하지 않고 새 편집자의 기기에 다시 만듭니다.

05 · Review

검수는 Draft PR 하나에서 반복한다

중간 확인과 최종 승인을 다른 PR로 나누지 않습니다. 첫 검수 요청에 Draft PR을 만들고, 수정과 재검수를 같은 대화 안에서 이어갑니다.

PR 상태와 반복 검수검수 라운드마다 PR을 새로 만들지 않음

댓글을 연결할 수 있는 범위

  • 렌더링 문서의 선택한 문장
  • diff 한 줄 또는 연속된 여러 줄
  • Mermaid·표·코드 블록 전체
  • 선택하지 않으면 문서 전체

초기 선택형 결정

  • 질문 한 개, 단일 선택
  • 선택지 2~5개
  • 선택 이유는 선택적으로 작성
  • 결정 확정과 종료

리뷰 대화는 GitHub PR에 남고, 시간이 지나도 유지해야 할 최종 결정만 OKF history나 문서 본문으로 승격합니다.

06 · Finish

확정은 Squash merge, 그다음 안전하게 정리한다

작업 중 checkpoint는 충분히 남기되, 기본 브랜치에는 하나의 의미 있는 변경으로 합칩니다.

1

최종 동기화 확인

로컬에만 남은 수정이 없는지 확인하고 마지막 checkpoint를 push합니다.

차단 조건동기화되지 않은 변경이 있으면 병합·정리를 진행하지 않습니다.
2

Ready for review → 승인

같은 PR을 최종 검수 상태로 전환하고 승인받습니다.

GitHub 원본댓글, 결정과 승인 이력이 PR에 유지됩니다.
3

Squash merge

여러 checkpoint commit을 main의 하나의 완결된 변경으로 합칩니다.

문서 결과병합된 OKF Markdown이 새로운 확정본이 됩니다.
4

Issue 종료와 로컬 정리

문서 변경 Issue를 닫아 잠금을 해제한 뒤 worktree와 로컬·원격 작업 브랜치를 정리합니다.

다른 기기병합 상태를 감지하면 해당 기기의 로컬 worktree를 별도로 정리합니다.
정상 완료

병합 후 정리

main 갱신 → Issue 종료 → 잠금 해제 → worktree와 작업 브랜치 제거 순서를 지킵니다.

작업 중단

기본은 원격 브랜치 보존

마지막 checkpoint를 push하고 중단 이유를 남긴 뒤 Issue·PR을 닫습니다. 완전 삭제는 별도 확인이 있을 때만 수행합니다.

07 · Offline

오프라인에서는 로컬 초안으로 편집하고, 재연결 뒤 안전성을 판정한다

오프라인 편집 자체를 막지 않습니다. 다만 그동안 원격 잠금을 확보했다고 간주하지 않고, 온라인이 된 뒤 GitHub 상태와 문서 변경을 비교합니다.

재연결 판단어떤 경우에도 로컬 초안을 조용히 덮어쓰지 않음
오프라인

잠금 소유권 없음

편집은 가능하지만 원격 Issue나 branch를 만들 수 없으므로 다른 팀원이 같은 문서를 고칠 가능성은 남아 있습니다.

로컬 메타데이터

복구에 필요한 최소 정보

document_id, base_commit, 로컬 worktree 경로, 시작 시각을 기기 설정에 저장합니다.

재연결

검증 후 공유

Issue 잠금, 현재 main, 같은 사용자의 원격 branch를 비교한 뒤에만 commit·push합니다. 다른 기기에서 같은 branch가 바뀌었다면 먼저 비교·해결하며 자동 덮어쓰기는 하지 않습니다.

08 · Exceptions

충돌과 비정상 상태에서는 자동으로 밀어붙이지 않는다

Hub의 잠금은 협업 규칙을 지키게 돕는 소프트 잠금입니다. GitHub에서 직접 바꾼 상태까지 강제로 막지는 않으므로 이상 상태를 감지하면 읽기 전용으로 전환합니다.

잠금 경쟁

동시에 Issue를 만든 경우

생성 전·후로 열린 문서 변경 Issue를 확인합니다. Hub가 동시에 만든 Issue끼리는 가장 낮은 Issue 번호가 이기고, 패자는 설명 댓글을 남긴 뒤 닫습니다.

수동 중복

기존 중복 Issue가 발견된 경우

Hub가 임의로 닫지 않습니다. 영향받은 문서를 읽기 전용으로 열고 사용자가 유지할 Issue를 선택하게 합니다.

여러 문서

잠금은 all-or-nothing

하나의 변경 작업에 포함된 문서 중 하나라도 다른 작업이 잠갔다면 전체 잠금 확보를 실패시킵니다. 일부만 잡고 시작하지 않습니다.

상태 불일치

Issue·PR·로컬 상태가 어긋난 경우

예: Issue는 닫혔지만 PR이 열려 있거나 로컬 수정이 남은 상태. Hub는 잠금 상태 확인 필요로 표시하고 복구 전까지 편집을 막습니다.

시간 만료로 잠금을 풀지 않습니다. 열린 문서 변경 Issue가 잠금의 원본입니다. 사용자가 병합하거나 명시적으로 중단할 때 해제합니다.

09 · Storage

결국 무엇이 어디에 저장되는가

DB 없이도 공유할 수 있는 이유는 정보의 성격에 따라 Git, GitHub, 로컬 저장소의 책임을 나눴기 때문입니다.

GitHub

팀 간 작업 소유권, 리뷰, 결정, 승인, 진행 상태

로컬 기기

자동저장, 로컬 경로, UI 설정, 오프라인 초안, 인증 토큰

하루 작업을 한 문장으로 요약하면
문서를 고치기로 한다 → 기존 문서면 Issue 잠금을 잡는다 → 독립 worktree에서 편집한다 → checkpoint를 push한다 → Draft PR에서 비동기 검수한다 → 같은 PR에서 수정과 결정을 반복한다 → Ready로 최종 승인받고 Squash merge한다 → Issue와 worktree를 정리한다.
