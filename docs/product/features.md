# 기능 정의

기능 영역은 화면과 일대일로 확정하지 않습니다. 하나의 기능이 여러 화면에서 사용될 수 있으며, 화면 정보 구조는 유스케이스가 승인된 뒤 결정합니다.

## 1. 워크스페이스 관리

**상태: 합의**

- 최근 워크스페이스 열기
- 새 워크스페이스 만들기
- 기존 OKF 지식 저장소 가져오기
- 백엔드·프론트엔드 코드 저장소 연결
- GitHub 계정과 동기화 상태 관리
- 기본 브랜치와 작업 규칙 설정

프로젝트 연결은 워크스페이스 내부의 주 탐색 트리가 아니라 시작 화면이나 우측 상단 설정에서 진행합니다.

## 2. Documents 작성과 탐색

**상태: 합의, 세부 모델 미결정**

- OKF Markdown 자동 렌더링
- 리치 문서 편집과 Markdown 원문 편집
- Mermaid 시퀀스 다이어그램 작성과 렌더링
- 특정 문서의 Git 커밋 이력과 diff
- 기능·시나리오·처리 케이스 탐색
- 분석, 테스트, 실험과 기술 조사 문서 작성
- 관련 코드, API, Issue, PR, 커밋 연결
- 현재 지식과 변경 제안 버전 구분

문서 타입과 제품 구조 트리의 기본 깊이는 추가 조사가 필요합니다.

## 3. 리뷰와 변경 제안

**상태: 후보 방향 합의**

- 기본 브랜치의 확정 문서에서 변경 제안 생성
- 제안 브랜치의 문서와 Mermaid 편집
- 기존 버전과 제안 버전 비교
- 문서 블록과 줄 단위 댓글
- 선택형 결정 요청
- 승인과 수정 요청
- 브랜치 갱신과 재검토
- 병합 후 확정 문서와 history 반영

리뷰 내용은 확정 문서에 바로 반영하지 않습니다. 승인된 변경만 기본 브랜치에 병합합니다.

리뷰는 독립 전역 화면을 소유하지 않습니다. 사용자가 처리해야 할 요청은 Home에서 모아 보여주며, 실제 비교·댓글·선택·승인은 대상 문서나 PR의 문맥 안에서 처리합니다.

## 4. GitHub Project와 구현 진행

**상태: 기본 구조 합의, 상세 필드 연동 미결정**

- 워크스페이스당 하나의 GitHub Project 연결
- 여러 저장소의 Issue와 PR을 모은 Board와 목록
- 나에게 할당된 Issue와 Milestone
- 담당자와 관련 기능 필터
- GitHub Issue 생성, 조회, 댓글과 종료
- 관련 기능·시나리오·처리 케이스·문서 연결
- 연결된 PR과 코드 확인

Project는 계획과 진행 상태를 표현하는 상위 화면이며, 구현 작업의 공유 원본은 GitHub Issue입니다. 초기 Project 항목은 실제 Issue와 PR로 제한하고 Draft Issue는 제외합니다. 사용자 시나리오 수준 진행 요약은 Home에서 문서 상태와 관련 Issue를 조합해 계산합니다.

## 5. Home과 활동 이력

**상태: 기본 구성 합의, 세부 상호작용 미결정**

- `프로젝트 진행 상황` 제목과 그 아래의 인사말·오늘 날짜
- 열린 Issue, 진행 중인 Issue, 이번 주 완료 Issue 요약
- 기능별 진행률과 `완료 Issue 수 / 전체 Issue 수`
- 내가 확인해야 하는 검토, 결정, 응답과 작업 시작 가능 항목
- 막힌 작업과 오래 대기 중인 항목
- 발생 시간을 포함한 설계·승인·구현·테스트·배포 활동
- 최근 열어본 항목과 즐겨찾기

백엔드 API 완료처럼 후속 작업자가 바로 일을 시작할 수 있는 이벤트는 `작업 시작 가능`으로 보여줍니다. Home은 진행 상황, 확인할 항목과 활동 타임라인을 모아 보여주지만 실제 문서 편집, 리뷰와 Issue 처리는 각 원래 문맥으로 이동해 수행합니다.

## 6. 문서와 코드 연결

**상태: 합의**

- 하나의 문서에서 여러 코드 파일과 심볼 연결
- 하나의 코드 위치에서 관련 문서와 처리 케이스 역탐색
- 브랜치, 커밋, PR과 문서 버전 연결
- Hub 안에서 코드 읽기, 검색과 diff
- IDE에서 해당 위치 열기

코드 수정, 빌드, 실행과 디버깅은 초기 범위에 포함하지 않습니다.

## 7. API 계약과 문서

**상태: 미결정**

- OpenAPI 명세 렌더링
- 기능과 처리 케이스에서 `operationId` 기반 API 연결
- 요청·응답 DTO와 오류 계약 확인
- API 변경 제안과 리뷰
- Postman 또는 별도 API Client와의 역할 분담

현재 추천 후보는 OpenAPI를 Git 원본으로 두고 Postman은 실행·테스트 도구로 사용하는 방식입니다. 기존 백엔드 API 문서화 방식 확인이 먼저 필요합니다.

## 8. MCP와 외부 AI

**상태: 방향 합의**

- 워크스페이스, 기능, 문서, 진행 상태와 대기 요청 조회
- 문서와 Mermaid 초안 작성
- 상태 변경과 코드 참조 연결
- 리뷰 또는 의사결정 요청 생성
- 변경 diff 확인
- 공유·병합·삭제 같은 작업에 사용자 확인 요구

초기에는 앱이 꺼져 있어도 사용할 수 있는 로컬 STDIO MCP 서버가 유력한 후보입니다.
