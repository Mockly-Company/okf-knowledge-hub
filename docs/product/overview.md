# 제품 개요

## 배경

백엔드 개발자와 프론트엔드 개발자 한 명씩 소규모로 작업하고 있습니다. 두 개발자의 작업 시간이 다르기 때문에 개발 도중 필요한 설계 확인과 의사결정이 주로 메신저에서 비동기적으로 이루어집니다.

기능을 구현하다 추가 요구사항이나 애매한 설계를 발견하면 관련 코드와 화면을 캡처하고, 상대방이 이해할 수 있도록 배경을 다시 설명해야 합니다. 답변을 받은 뒤에도 결정 결과와 이유가 실제 문서나 코드에 자동으로 연결되지 않아 시간이 지나면 맥락을 찾기 어렵습니다.

Notion, Postman, GitHub Issue/PR과 메신저를 함께 사용하면 각각은 유용하지만 소규모 팀에서는 작업 정보가 지나치게 여러 곳으로 분산됩니다. 그 결과 프로젝트 진행 상황이 잘 보이지 않고, 확인되지 않은 항목 때문에 작업이 자연스럽게 밀리는 문제가 생깁니다.

## 제품 목표

`okf-knowledge-hub`는 다음을 목표로 합니다.

- 개발 중 발생한 맥락을 별도로 다시 정리하지 않아도 상대 개발자에게 전달한다.
- 설계, 리뷰, 결정, 코드, 구현 상태와 이력을 하나의 작업 흐름으로 연결한다.
- 서로 다른 시간에 접속해도 현재 상태와 내가 처리할 일을 빠르게 파악한다.
- 사람이 편한 화면으로 작성한 결과를 AI와 자동화 도구가 이해할 수 있는 구조로 보존한다.
- GitHub 화면을 직접 오가지 않고도 Issue/PR 기반 작업을 수행한다.

## 주요 사용자

- 백엔드 개발자
- 프론트엔드 개발자
- 선택적으로 MCP를 통해 접근하는 Codex 등의 AI 도구

작성자와 검토자는 고정된 역할이 아닙니다. 같은 사용자가 상황에 따라 설계를 작성하거나 상대방의 요청을 검토합니다.

## 제품 원칙

### 작업 과정에서 문서가 갱신되어야 한다

개발이 끝난 뒤 별도로 문서를 작성하는 대신, 기능을 설계하고 리뷰하고 구현하는 과정에서 지식이 자연스럽게 누적되어야 합니다.

### 현재 지식과 변경 제안을 구분한다

기본 브랜치의 문서는 팀이 현재 합의한 지식입니다. 아직 검토 중인 Mermaid, API 계약과 문서 변경은 별도의 브랜치와 PR에서 제안 상태로 유지합니다.

### 한 화면은 한 저장소를 뜻하지 않는다

사용자는 Hub 안에서 문서, 코드, Issue, PR과 리뷰를 함께 보지만, 각 정보의 공유 원본은 Git과 GitHub에 유지할 수 있습니다.

### 트리를 모든 정보의 저장 구조로 사용하지 않는다

제품 구조 트리는 기능의 계층만 표현합니다. 분석 문서, API, 결정, 코드와 리뷰는 별도의 자료이며 여러 기능이나 처리 케이스에 연결될 수 있습니다.

### 로컬 우선으로 작게 시작한다

초기에는 중앙 서버와 SQLite 같은 별도 애플리케이션 데이터베이스를 두지 않습니다. 로컬 앱, Git과 GitHub를 조합해 두 명의 실제 협업 문제부터 해결합니다.

팀이 공유하는 워크스페이스 정의는 OKF Git 저장소의 `.okf/workspace.yml`에 둡니다. 기기별 checkout 경로와 UI 설정만 로컬 설정 파일에 저장하고 인증 정보는 OS 보안 저장소에 둡니다. 초기에는 OKF 저장소 하나를 워크스페이스 하나로 취급하며 앱은 여러 워크스페이스 전환 기능을 제공하지 않습니다.

## 초기 범위가 아닌 것

- 완전한 Jira 대체
- 완전한 Confluence 또는 Notion 대체
- IDE 수준의 코드 편집, 실행과 디버깅
- 별도 요금이 필요한 내장 AI 작성 기능
- 모든 팀에 맞는 범용 프로젝트 관리 플랫폼
